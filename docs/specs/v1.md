# Spécifications v1 — ISBibliotheque

## 1. Architecture applicative

### Principe

Les applications ne sont pas codées en dur dans le frontend.  
Le portail découvre dynamiquement une liste d'applications via un **backend API** qui lit un répertoire de manifests.

```
infra/apps.registry/
  ├── production/
  │   ├── metadata.json       # nom, description, icône, catégorie, type d'accès
  │   └── docker-compose.yml  # (optionnel) si app à lancer via Docker
  ├── stocks/
  │   ├── metadata.json
  │   └── docker-compose.yml
  └── facturation/
      └── metadata.json       # redirect simple, pas de compose
```

### Flux

```
Frontend  ←API→  Backend (Express/NestJS)  ←lecture→  infra/apps.registry/
                          │
                          ├── Redirect → ouvre URL dans un nouvel onglet
                          └── Docker   → POST /api/apps/:id/start
                                         → docker compose up -d
                                         → retourne l'URL locale (port mapping)
```

## 2. Format d'un manifest d'application

**`metadata.json`** (dans chaque dossier du registry) :

```json
{
  "$schema": "../../schemas/app-manifest.json",
  "id": "production",
  "name": "Suivi de production",
  "description": "Ordres de fabrication, avancement et alertes temps réel",
  "category": "Production",
  "icon": "Zap",
  "access": {
    "type": "redirect",
    "url": "https://erp.isb-group.fr/production"
  },
  "roles": ["production.manager", "production.operator"]
}
```

Pour une app Docker :

```json
{
  "id": "custom-reporting",
  "name": "Reporting personnalisé",
  "description": "Générateur de rapports qualité",
  "category": "Qualité",
  "icon": "BarChart3",
  "access": {
    "type": "docker",
    "composeFile": "docker-compose.yml",
    "serviceName": "app",
    "internalPort": 3000
  },
  "roles": ["admin", "quality.manager"]
}
```

### Champs

| Champ                 | Type                   | Description                                                                           |
| --------------------- | ---------------------- | ------------------------------------------------------------------------------------- |
| `id`                  | string                 | Identifiant unique (slug)                                                             |
| `name`                | string                 | Nom affiché dans la grille                                                            |
| `description`         | string                 | Description courte                                                                    |
| `category`            | enum                   | `Gestion`, `Production`, `RH`, `Finance`, `Qualité`, `Logistique`, `Commercial`, `IT` |
| `icon`                | string                 | Nom de l'icône Lucide                                                                 |
| `access.type`         | `redirect` ou `docker` | Mode d'accès                                                                          |
| `access.url`          | string                 | (redirect) URL cible                                                                  |
| `access.composeFile`  | string                 | (docker) Chemin relatif du docker-compose                                             |
| `access.serviceName`  | string                 | (docker) Nom du service dans le compose                                               |
| `access.internalPort` | number                 | (docker) Port interne du service                                                      |
| `roles`               | string[]               | Rôles autorisés (vide = tous)                                                         |

## 6. Définition "Done" v1

- [ ] **Registry lisible** — `infra/apps.registry/` créé avec au moins 3 manifests
- [ ] **API backend** — endpoint `GET /api/apps` fonctionnel, retourne les apps du registry
- [ ] **Frontend branché sur l'API** — la grille affiche les vraies apps (plus de mock statique)
- [ ] **Filtres** — catégorie + recherche fonctionnels
- [ ] **Redirect** — clic sur une app `redirect` → ouvre l'URL dans un nouvel onglet
- [ ] **Docker start/stop** — API `POST /start` et `POST /stop` pour les apps de type `docker`
- [ ] **Déploiement** — tout tourne dans Docker (frontend + backend)
- [ ] **Auth** — page de login avec récupération du rôle utilisateur
- [ ] **403** — écran Non autorisé si l'utilisateur n'a pas le rôle requis
- [ ] **Loading + Error** — écrans de chargement et d'erreur branchés
- [ ] **Lint + build** — CI passe

## 7. Risques identifiés

| Risque                                                             | Impact   | Mitigation                                                                                        |
| ------------------------------------------------------------------ | -------- | ------------------------------------------------------------------------------------------------- |
| Socket Docker non accessible depuis le backend                     | Bloquant | Monter `/var/run/docker.sock` dans le conteneur backend (ou utiliser Portainer API)               |
| Ports dynamiques des apps Docker non routables                     | Moyen    | Assigner des ports hôtes fixes dans chaque docker-compose, ou utiliser un reverse proxy (Traefik) |
| Réseau interne inaccessible depuis le navigateur                   | Bloquant | Tunnel VPN ou reverse proxy accessible                                                            |
| Certificats SSL manquants (apps internes en HTTP)                  | Moyen    | Terminaison SSL au niveau du reverse proxy                                                        |
| CORS entre le portail et les apps tierces                          | Moyen    | Reverse proxy unifié ou proxy_pass côté backend                                                   |
| Perf : latence si beaucoup de apps Docker                          | Faible   | Cache des statuts avec refresh périodique                                                         |
| Sécurité : un utilisateur pourrait lancer n'importe quel conteneur | Bloquant | Validation des rôles côté backend avant chaque `start`                                            |
| Corruption du registry si écriture concurrente                     | Faible   | Registry en lecture seule côté backend ; écriture réservée aux admins via déploiement             |

## 8. Stack technique proposée

| Couche        | Technologie                               | Raison                                  |
| ------------- | ----------------------------------------- | --------------------------------------- |
| Frontend      | React 18 + Vite + TypeScript              | Existant                                |
| Backend       | Express.js (ou Hono) + TypeScript         | Léger, rapide à mettre en place         |
| Registry      | Fichiers JSON dans `infra/apps.registry/` | Pas de base de données nécessaire en v1 |
| Docker        | API Docker via `dockerode`                | Contrôle des conteneurs depuis Node.js  |
| Orchestration | docker-compose pour le déploiement global | Standard, simple                        |
| Auth          | Entra ID (passport-azure-ad)              | Déjà utilisé chez ISB                   |
| Reverse proxy | Nginx ou Traefik                          | Routage unifié                          |
